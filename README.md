# Каталог Сервис.com — сайт с API-выгрузкой из 1С

Полноценное веб-приложение: каталог товаров, публичный API, **API для выгрузки из 1С**,
админ-панель с массовым изменением цен. Стек: Node.js + Express + SQLite.

---

## ⚠️ ЧТО НУЖНО ДЛЯ ЗАПУСКА (передать разработчику)

Это НЕ статический сайт — ему нужен **сервер с Node.js** (не подойдёт бесплатный Netlify-drop).
Разворачивается на: Render.com, Railway.app, Fly.io, или любой VPS (Timeweb, Beget, Selectel и т.п.).

Требуется специалист для: развёртывания на хостинге и настройки выгрузки на стороне 1С.

---

## СТРУКТУРА
```
server.js          — сервер (API + раздача каталога и админки)
db.js              — база данных SQLite (схема)
scripts/seed.js    — загрузка стартовых 270 товаров
public/            — сайт-каталог (то, что видят клиенты)
admin/             — админ-панель
.env.example       — пример настроек (скопировать в .env)
```

## ЗАПУСК ЛОКАЛЬНО
```bash
npm install
cp .env.example .env       # отредактировать пароли/токены
npm run seed               # загрузить стартовый каталог (один раз)
npm start                  # сервер на http://localhost:3000
```
- Каталог: http://localhost:3000
- Админка: http://localhost:3000/admin/  (пароль из .env → ADMIN_PASSWORD)

## НАСТРОЙКИ (.env)
- `ADMIN_PASSWORD` — пароль входа в админку
- `JWT_SECRET` — длинная случайная строка (подпись сессий)
- `IMPORT_TOKEN` — секретный токен для выгрузки из 1С
- `PORT` — порт (на хостинге обычно задаётся автоматически)

## 🔌 ВЫГРУЗКА ИЗ 1С (главное)

1С (или промежуточный скрипт) отправляет товары на эндпоинт:

```
POST  https://ВАШ-САЙТ/api/import
Заголовок:  Authorization: Bearer <IMPORT_TOKEN>
Тело (JSON):
{
  "source": "1С УТ",
  "products": [
    {
      "sku": "DH-IPC-HFW1230S",     // артикул — КЛЮЧ (по нему обновление)
      "brand": "Dahua",
      "model": "DH-IPC-HFW1230S",
      "group": "Видеонаблюдение",    // направление
      "cat": "Цилиндрическая камера",
      "desc": "2 Мп уличная IP-камера...",
      "res": "2 МП",
      "price": 28000,                 // РРЦ
      "stock": 14,                    // остаток
      "img": "dahua-hfw1230s.png"     // имя файла в /public/images ИЛИ URL картинки
    }
  ]
}
```
Ответ: `{ "ok": true, "received": N, "created": X, "updated": Y }`

Логика: товар ищется по `sku`. Есть — обновляется (цена, остаток, описание). Нет — создаётся.
Так каталог поддерживается в актуальном состоянии автоматически.

### Как 1С отправляет данные
Варианты (выбирает 1С-специалист):
- HTTP-запрос прямо из 1С (объект `HTTPСоединение`, по расписанию регламентным заданием);
- Промежуточный скрипт: 1С выгружает JSON/CSV → скрипт шлёт на /api/import;
- Обмен через файл + утилита загрузки.

Рекомендация: регламентное задание в 1С раз в сутки/час шлёт прайс с остатками на /api/import.

## АДМИН-ПАНЕЛЬ
- Вход по паролю.
- Редактирование товаров (цена, описание, акция, остаток, фото) — сохраняется на сервере сразу.
- **Массовое изменение цен**: процент (+/−), опционально по бренду/направлению, округление.
- Акции (🔥): галочка + старая цена → товар попадает в ленту «Горячие предложения».
- Журнал выгрузок из 1С.

## ФОТО
- Файлы лежат в `public/images/`. В товаре поле `img` = имя файла или полный URL.
- При выгрузке из 1С можно слать прямые URL картинок — тогда файлы на сервер класть не нужно.

## БЕЗОПАСНОСТЬ
- Смените `ADMIN_PASSWORD`, `JWT_SECRET`, `IMPORT_TOKEN` перед публикацией.
- API-выгрузка защищена токеном; админка — паролем и токеном сессии.
- На проде включите HTTPS (на Render/Railway он есть из коробки).

## РАЗВЁРТЫВАНИЕ (кратко, для разработчика)
1. Залить репозиторий на GitHub.
2. Render.com → New Web Service → подключить репозиторий.
3. Build command: `npm install`; Start command: `npm start`.
4. Указать переменные окружения (ADMIN_PASSWORD, JWT_SECRET, IMPORT_TOKEN).
5. После первого деплоя выполнить `npm run seed` (или загрузить товары через выгрузку из 1С).
6. Привязать домен servis-com.kz.
7. SQLite-файл хранится на диске сервиса — на Render подключить Persistent Disk,
   либо для production перейти на PostgreSQL (схема легко переносится).






## 🚀 БЫСТРАЯ ИНДЕКСАЦИЯ В ПОИСКЕ (SEO)
Реализованы приёмы для скорейшего попадания в Яндекс и Google:

### 1. IndexNow — мгновенное уведомление (главное)
Сайт сам сообщает Яндексу/Bing о новых и изменённых товарах — индексация за часы, а не недели.
- Сгенерируйте ключ (32 hex-символа) и задайте `INDEXNOW_KEY` в .env.
- Сервер отдаёт файл-ключ `/<ключ>.txt` (проверка владения) и пингует IndexNow при выгрузке из 1С и изменениях в админке.
- Работает автоматически. Ничего нажимать не нужно.

### 2. Подтверждение прав в вебмастерах
- **Яндекс.Вебмастер** (webmaster.yandex.ru) → добавьте сайт → способ «Мета-тег» → скопируйте значение content → в .env `YANDEX_VERIFICATION=...`
- **Google Search Console** (search.google.com/search-console) → аналогично → `GOOGLE_VERIFICATION=...`
- Сервер подставит мета-теги на все страницы. После проверки — добавьте sitemap: `https://ваш-домен/sitemap.xml`.

### 3. Что уже настроено
- **sitemap.xml** — динамический, со всеми товарами, датами изменения (`lastmod`) и приоритетами. Обновляется сам.
- **robots.txt** — разрешает индексацию, закрывает админку/API, указывает sitemap.
- **LocalBusiness-разметка** на главной (адрес, телефон, гео, часы) — для локальной выдачи и Яндекс/Google Карт.
- **Product + BreadcrumbList** на каждом товаре — rich snippets (цена, наличие в выдаче).
- **Canonical, Open Graph** на всех страницах.

### Первые шаги после запуска (чек-лист):
1. Задать `SITE_URL`, `INDEXNOW_KEY`, `YANDEX_VERIFICATION`, `GOOGLE_VERIFICATION` в .env.
2. Подтвердить сайт в Яндекс.Вебмастере и Google Search Console.
3. Добавить sitemap в обоих сервисах.
4. Зарегистрировать компанию в **Яндекс.Бизнес** и **Google Business Profile** (для локальной выдачи и карт — для B2B в одном городе это даёт больше, чем чистый SEO).

## 📈 ЯНДЕКС.МЕТРИКА (аналитика)
Счётчик встроен во все страницы (каталог + страницы товаров). Чтобы включить:
1. Зарегистрируйте счётчик на https://metrica.yandex.ru (получите номер, напр. 98765432).
2. Задайте в .env: `YM_ID=98765432`.
3. Сервер сам подставит номер во все страницы. Без `YM_ID` счётчик не активируется (данные не шлёт).
Включены: clickmap, отслеживание ссылок, точный показатель отказов, вебвизор.
**Цели (события) уже настроены в коде** — в Метрике создайте цели типа «JavaScript-событие» с идентификаторами:
- `add_to_cart` — товар добавлен в заявку;
- `order_sent` — заявка отправлена.
Так вы увидите конверсию: сколько посетителей → добавили → отправили заявку.

## 🔒 БЕЗОПАСНОСТЬ И НАДЁЖНОСТЬ (реализовано)
- **Хэш пароля** (scrypt, без хранения в открытом виде). Сгенерировать хэш: `npm run hash "пароль"` → в `ADMIN_PASSWORD_HASH`.
- **Rate-limiting**: вход в админку (10 попыток / 15 мин), импорт (120 / 15 мин), общий API (300 / мин) — защита от брутфорса.
- **Защита от дефолтных секретов**: в `NODE_ENV=production` сервер НЕ запустится с паролями по умолчанию.
- **Валидация выгрузки**: лимит количества (`MAX_IMPORT`), обрезка длинных строк, приведение типов, мусорные записи пропускаются (в ответе `skipped`).
- **Helmet** (защитные HTTP-заголовки), **CORS** (настраивается через `CORS_ORIGINS`), **gzip-сжатие** (быстрая отдача каталога), кэш-заголовки на каталог и картинки.
- **Безопасное сравнение** токенов (защита от тайминг-атак).
- **Обработка ошибок**: глобальный error-handler, без утечки стектрейсов клиенту; логирование в консоль.
- **Health-check**: `GET /health` (для мониторинга хостинга).

## 🔍 ВТОРОЙ ПРОХОД РЕВЬЮ (доп. улучшения)
- **Склад не светится наружу**: публичный API отдаёт только `inStock` (в наличии/нет), точные остатки — лишь в админке. Защита коммерческих данных от конкурентов.
- **Загрузка фото из админки**: кнопка ⬆ в строке товара — выбрать файл, он загрузится на сервер (`POST /api/admin/upload`, base64, до 3 МБ, png/jpg/webp, имя санитизируется от path-traversal).
- **Логирование**: morgan (access-log всех запросов) + предупреждения о неудачных входах в админку (для мониторинга брутфорса).
- **Корректный 404**: неизвестные API-пути → JSON; обычные URL → каталог (deep links не ломаются).
- **Docker**: `Dockerfile` + `.dockerignore` для воспроизводимого развёртывания. `.nvmrc` (Node 20).
- **Тесты**: `npm test` (tests/api.test.js) — интеграционные проверки безопасности и логики.

### Осознанно НЕ реализовано (чтобы не переусложнять под реальную нагрузку):
- Строгий CSP против XSS — потребовал бы переписать весь фронтенд (inline-обработчики); риск на каталоге без аккаунтов/платежей низкий.
- PostgreSQL/очереди/CDN — избыточно для локального бизнеса; SQLite достаточно. Переход на PG — только при реальном росте нагрузки (поменять db.js).
- Отзыв/refresh JWT — для одного администратора 12-часового токена достаточно.
Это технический долг «на будущее», а не блокеры запуска.


## 🧪 ТРЕТИЙ ПРОХОД (командное ревью) — доп. исправления
- **Исправлен хранимый XSS**: все поля товаров экранируются при отрисовке (каталог и админка). Вредоносный текст в названии/описании (из 1С или админки) выводится как текст, скрипт не выполняется.
- **Upload проверяет содержимое файла** (magic bytes PNG/JPEG/WebP), а не только расширение — нельзя залить переименованный не-картинку.
- **Graceful shutdown**: при SIGTERM/SIGINT сервер закрывает соединения и базу корректно (важно при передеплое).
- **Docker укреплён**: `npm ci` по lockfile (воспроизводимость), запуск от непривилегированного пользователя, HEALTHCHECK. Добавлен `package-lock.json`.

### Известный технический долг (не блокеры, на будущее):
- A11y фронтенда (ARIA, навигация с клавиатуры), индикатор загрузки каталога.
- Виртуализация списка при 2000+ товаров (сейчас рендерятся все карточки сразу).
- Строгий CSP (требует переписать inline-обработчики; риск низкий — каталог без аккаунтов/платежей).
- scryptSync/синхронная запись файла блокируют event loop при высокой нагрузке (для текущего масштаба несущественно).
Все пункты — оптимизации под рост, а не дефекты, мешающие запуску.


### Ревью страниц товаров (доп. проход)
- **Закрыт XSS в JSON-LD/инлайн-данных**: данные товара (название, описание) сериализуются безопасно (`<`,`>`,`&` экранируются), чтобы вредоносный текст из 1С/админки не мог разорвать тег `<script>` микроразметки. Проверено на payload `</script><script>...`.
- HTML-контекст (заголовок, характеристики, похожие) — экранируется `esc()`. SQL — параметризован. Ссылки — `encodeURIComponent`.

## ПРИМЕЧАНИЕ ПРО БАЗУ
По умолчанию SQLite (файл data.sqlite) — просто и без отдельного сервера БД.
Для нагруженного прода рекомендуется PostgreSQL (поменять только db.js).
