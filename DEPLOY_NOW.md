## 🔴 Срочно: сайт лежит на проде — этот батч его оживит

**Причина — дефект кода, а не ваша настройка.** Дефолт домена в коде был `https://s1com.kz`, и host-редирект включался САМ, даже без `SITE_URL` в Render. Весь сайт уходил в 301 на неподключённый домен; живыми оставались только `/health` и `/api/`, поэтому со стороны API всё выглядело здоровым.

Исправлено (проверено живьём — сервер в prod-режиме без `SITE_URL` отдаёт `/` → 200):
- дефолт домена → `https://servis-catalog.onrender.com` (рабочий адрес);
- host-редирект включается **только при явно заданном `SITE_URL`** (`server.js:418`).

**Что делать:** просто выложить батч — сайт оживёт сам, трогать ENV не нужно.
**Оживить прямо сейчас, до деплоя:** Render → `servis-catalog` → Environment → `SITE_URL=https://servis-catalog.onrender.com` → Save (либо `NO_HOST_REDIRECT=1`).
**Когда домен купят и делегируют:** задать `SITE_URL=https://s1com.kz` (и то же на трёх cron-сервисах) — редирект со старого хоста включится сам.

### 🔴 Закрыт XSS через данные поставщика (критично)

Поля товара (`brand`, `model`, `img`, `images`) приезжают из **чужого API** (Al-Style/Complex) и вставлялись в страницу без экранирования в 4 местах. Одно «грязное» значение в фиде поставщика = выполнение чужого скрипта на витрине (угон сессии админа, подмена контактов, редирект на фишинг).

Починено: `lib/product-page.js` (галерея + `window.PRODUCT`/`PAGE_GROUP`), `lib/category-page.js` (`window.PAGE_BRAND`), `server.js` (внешний URL картинки в `ssrGrid` на 7 страницах разделов). Доказано пробой: на старом коде — 4 провала, на новом — чисто.

### 🟠 Четыре дефекта склейки дублей

Все воспроизведены и закрыты, покрыты новым `tests/integration/match-fixes.test.js` (18 проверок; на старом коде тест краснеет — проверено):

1. **Товар исчезал с витрины целиком.** Если «победителя» в очереди ранее спрятали руками (bulk-hide — ровно тот воркфлоу, что советовали раньше), `resolve` не возвращал его на витрину, но прятал всех остальных. Ответ API — «успешно». Теперь такой выбор отклоняется с внятной ошибкой.
2. **«Это разные товары» не запоминалось** — та же группа возвращалась в очередь каждый прогон (с `AUTO_MERGE` — ~48 раз в сутки). Теперь решение залипает (`status='dismissed'`, дедуп по ключу группы). Появилась кнопка **«🚫 Не дубли»** — посмотреть отменённое и вернуть в очередь, если передумали.
3. **«Показать» склеенный товар не переживал импорт** — снимало `hidden_manual`, но не `merged_into`, и ночной импорт прятал товар снова. Зеркальная ошибка к прошлому фиксу «скрытие залипает».
4. **Удаление победителя хоронило проигравших** — они оставались скрытыми со ссылкой на мёртвый id, импорт их не возвращал. Теперь `delete` сначала освобождает склеенных в него.

## ⚠️ Важно знать: `AUTO_MERGE` включать НЕ нужно

Проверено на реальном коде: **при нынешней паре поставщиков надёжных (EAN) склеек не бывает вообще.** Надёжная группа требует совпавший штрихкод у **двух разных** поставщиков, а у API Complex штрихкода нет в принципе. Значит:

- кнопка «🔗 Склеить надёжные (EAN)» и `AUTO_MERGE=true` **не склеят ни одной позиции**;
- все дубли идут в **очередь спорных** на ваш ручной разбор — это нормальный рабочий режим;
- `AUTO_MERGE=true` даст только лишний полный скан каждые 30 минут. **Оставьте выключенным** (так и стоит в `render.yaml`).

EAN-ветка окупится, когда появится третий поставщик, отдающий штрихкоды. Прошлый `DEPLOY_NOW` обещал обратное — это было ошибочное ожидание.

## ENV в дашборде Render (задать руками)

1. `ADMIN_PASSWORD` — обязательно, иначе сервер не стартует.
2. `SITE_URL` — **пусто или `https://servis-catalog.onrender.com`**. `https://s1com.kz` — только после делегирования домена, одновременно на всех трёх cron.
3. `ALSTYLE_API_KEY` — на web (иначе диагностика Al-Style 400) и на обоих alstyle-cron.
4. `COMPLEX_API_KEY` — на web (для пикера брендов Complex) и на `complex-catalog`.
5. Опционально: `YM_ID`, `YANDEX_VERIFICATION`, `GOOGLE_VERIFICATION`, `INDEXNOW_KEY`.
6. Проверить, что диск `data` примонтирован на `/data` (Render → Disks).
7. `AUTO_MERGE` — **не включать** (см. выше).

⚠️ `ALSTYLE_API_KEY`/`COMPLEX_API_KEY` на веб-сервисе не объявлены в `render.yaml` (стоят только в дашборде) — при пересоздании сервиса из blueprint потеряются.
